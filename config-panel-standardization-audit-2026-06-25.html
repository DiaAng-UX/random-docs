A standardization review of every right-hand configuration panel a designer touches while moving between tools — UI Designer, Process Designer, and the Data Model builder. The goal: one predictable panel grammar, so muscle memory carries from one surface to the next.

Part 0 — Executive summary — the friction, ranked

The eight inconsistencies that most hurt a user moving between panels, ranked by how badly they break transferred expectations (and, for the top items, trust in whether work was saved).

  1. Trust / data-lossThree different “autosave” delays, and only one panel tells you it saved. Data Model commits ~100 ms after you stop typing, Process Designer ~700 ms, UI Designer ~1000 ms — and only Process Designer shows a saved/saving indicator. Everywhere else you can’t tell whether an edit stuck before you click away. data-model…:296 · node-config…:256 · config-panel…:376
  2. Relearn per toolThe “use a static value vs. an expression” idea has three unrelated controls. Process actions use a parameter-type dropdown + code editor; Data Model validators use an f(x) computed-toggle; UI Designer binds via a separate expressions editor. No shared “switch this field to dynamic” affordance, and a bound field looks different in each. action-edit-param.component.ts · data-model-attribute-validations.component.ts
  3. Disorientation“Nothing selected” looks different in all three. Process Designer shows a helpful “Please select a node…” message; UI Designer shows a dedicated empty-state component; Data Model simply makes the whole panel vanish — no guidance, and the canvas appears to lose a feature. node-details.component.html:117 · config-panel.component.html:109 · data-model-builder.component.html
  4. Broken expectationMulti-select works in exactly one panel. Select several components in UI Designer and you get a combined list; do the “same” thing in Process Designer or Data Model and only one item (or nothing) responds. config-panel.component.html:44-50
  5. Visual whiplashThe same data type is drawn with different controls. A boolean is a switch in Process Designer (“Can go back”) but a checkbox in Data Model (“Sensitive data”, “Use in reporting”). Same meaning, different control, different click target. node-config.component.html:47 · data-model-attribute-details.component.html:248,266
  6. No shared anatomyPanels disagree on width, resizing, tabs, and header. UI Designer: fixed 344 px, tabbed (Settings / Styles), header shows the component name. Process Designer: resizable 270 px–50% viewport, tabbed (Node Config / Boundary / Actions / Gateway Decisions), header hardcoded “Node Settings”. Data Model: fixed width, no tabs, flat form, header is the entity name. config-panel.scss · side-panel-prototype.component.ts:122-125
  7. TerminologyOne field, two names, inside a single tool. The Data Model name field is labelled “Type name” at project level and “Object name” at process level — the divergence you flagged as a finding in itself. data-model-details-panel.component.ts:197-199
  8. Even the experiment isn’t consistentThe prototype’s own A/B toggle has three labels and two button orders. “Prototype settings” (Process & UI Designer) vs “Prototype · Validator editor” (Data Model); and Data Model lists Inline · Overlay while the others list Overlay · Inline. side-panel-prototype.component.html:3-36 · data-model-details-panel.component.html:5-38

1.5 Conditional fields

All three panels show/hide fields based on what’s selected — this is the one dimension where the mechanism is broadly similar even though the rules differ.

  • UI Designer swaps whole field groups by context and platform: Process-UI-only analytics & navigation config vs UI-Flow-only root title / parameters / conversation context; mobile-only container & scroll props (config-panel.store.ts:415-508).
  • Process Designer reveals tabs and sections by node type — “Gateway Decisions” only for exclusive gateways, boundary-event config only when relevant, “Can go back” only for User Tasks (node-details.component.html:28-95, node-config.component.html:43-83).
  • Data Model reveals type-dependent controls — array sub-type, enum reference, number type, decimal points, timezone — as the attribute Type changes (data-model-attribute-details.component.ts:213-325).
User impact: Generally good and expected. The risk is discoverability: fields appearing/disappearing without animation or grouping can feel like the panel “jumped.” inference — whether the reveal is animated or instant can’t be judged from static code; needs live observation.

Part 3 — Migration plan — sequenced by (user impact × effort)

Effort is estimated in plain terms from what the code showed during the audit (component count, shared-ness, whether a thing already exists in one panel). “Unlocks” notes which changes make later ones cheaper.

Phase 1 · Save-status cue everywhere

Quick win

Lift the Process Designer autosave indicator (idle / saving / saved) into a shared header component and mount it in UI Designer and Data Model. The control already exists and is design-system-styled.

  • Impact: high — directly fixes the #1 trust/data-loss finding.
  • Unlocks: a shared panel-header component (needed by Phase 4).
  • Evidence base: side-panel-prototype.component.html:77-106, AutosaveStatusService.

Phase 2 · Align debounce + standardize booleans + empty state

Quick win

Three small, independent unifications: (a) pick one debounce delay and apply it in all three commit engines; (b) switch Data Model booleans from checkbox to switch (or settle the reverse — Open Q); (c) reuse one empty-state component so Data Model stops hiding its panel.

  • Impact: medium-high; low risk, each is a localized change.
  • Evidence base: …:296/256/376; attribute-details…:248,266; config-panel.component.html:109.
Open question Which boolean control is canonical — switch or checkbox? Decide before touching the field.

Phase 3 · Unify validation placement

Moderate

Adopt inline-error-under-field on touch as the standard, and add the sticky “save blocked” summary so Process Designer stops silently halting autosave. Touches the validation rendering of each panel but the Data Model pattern is a ready template.

  • Impact: medium-high — removes a genuinely confusing silent-failure case.
  • Unlocks: consistent error model that the binding work (Phase 5) can lean on.

Phase 4 · Shared panel shell (header + sections + resize)

Large

Extract one panel shell: header grammar, context row, collapsible fixed-order sections, and resizable width with a shared min/default. Re-home each panel’s body into it. Highest structural payoff but it re-frames three components.

  • Impact: high — this is the “feels like one product” change.
  • Depends on: Phase 1 (header component) · Unlocks: consistent placement for Phases 3 & 5.
  • Evidence base: diverging roots config-panel.component.html, side-panel-prototype.component.html, data-model-details-panel.component.html.
Open question Tabs vs. stacked sections as the canonical grouping (Wireframe A).

Phase 5 · Unified ƒx binding control

Large

The deepest change: one ƒx toggle + shared expression editor + consistent bound-field marking, replacing three mechanisms. High value but touches data shapes (computed flags, param types) and the Monaco integration.

  • Impact: high; effort: large (data-model + editor work across React and Angular).
  • Best done last — benefits from the shared shell (Phase 4) and error model (Phase 3) already in place.

Phase 0 · Tidy the prototype A/B toggle

Quick win

Independent of the rest: give the “Prototype settings” toggle one label and one button order across all three panels (or gate it out of shipping builds). Trivial, and stops the experiment from being a visible inconsistency while the migration runs.

Sequencing note: do Phase 0 + 1 + 2 first (all quick wins, high trust payoff), then 3, then the large 4 → 5 pairing.

Part 4 — Measurement plan — Mixpanel, baseline before you change anything

Every proposed standard should be backed by data: baseline the current pain before migration, then prove each change worked. Below: a naming convention, the event catalog, the properties, and which events to instrument first.

Naming convention

Pattern: Config Panel — <Object> <PastTenseAction>  (Title Case, “Config Panel —” namespace prefix).

  • One namespace (Config Panel) so all panel analytics filter together regardless of tool.
  • The tool is a property (panel_area), never baked into the event name — so you compare across tools without string-matching.
  • Past-tense verbs (Opened, Edited, Applied) — events are facts that already happened.

Shared property on every event: panel_area ∈ {ui_designer_process_ui, ui_designer_ui_flow, process_designer, data_model_project, data_model_process} · plus resource_id, selection_type, app_id.
